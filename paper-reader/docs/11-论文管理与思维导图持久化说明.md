# 论文管理与思维导图持久化说明

## 1. 思维导图持久化

思维导图生成结果现在会保存为`backend/data/mindmaps/{paper_id}.json`。缓存内容包含中心主题、主要分支、子要点和Agent Trace ID。

接口行为如下：

- `GET /api/papers/{paper_id}/mind-map`：读取已经生成的缓存；不存在时返回404；
- `POST /api/papers/{paper_id}/mind-map`：优先复用缓存，不重复调用模型；
- `POST /api/papers/{paper_id}/mind-map?refresh=true`：忽略缓存，重新调用论文理解Agent生成并覆盖旧结果。

前端打开思维导图页面时会自动检查缓存。如果已经生成过，页面直接恢复中心主题和所有分支；点击“重新生成”才会使用`refresh=true`。

真实论文验收结果：首次DeepSeek生成约7.1秒，GET缓存约27毫秒，再次POST复用缓存约3毫秒，三次返回的Trace ID完全一致。缓存文件约2KB。

## 2. 论文删除管理

论文列表中增加“删除”按钮。用户确认后调用`DELETE /api/papers/{paper_id}`，后端按以下范围清理：

1. 上传的原始PDF；
2. MinerU提取的图片、图表、表格等资源目录；
3. JSON论文记录；
4. 结构化切片和检索索引；
5. 思维导图缓存；
6. 运行时内存知识库。

删除前对上传目录和资源目录进行绝对路径校验，防止越界删除。正在解析的论文不能删除，接口返回HTTP 409，避免后台任务继续写入已删除记录。前端也会禁用解析中论文的删除按钮，并在正式删除前显示确认对话框。

## 3. 回归测试

新增`scripts/test_persistence_and_delete.py`，测试全部在隔离临时目录中执行，不操作正式论文库。测试覆盖：

- 思维导图JSON写入和读取；
- POST接口复用缓存，不调用模型；
- 删除后PDF、索引、缓存和知识库切片全部消失；
- 删除后论文接口返回404；
- 解析中论文删除返回409并保留记录。

测试输出：`PERSISTENCE_DELETE_OK cache=reused delete=clean parsing=protected`。
