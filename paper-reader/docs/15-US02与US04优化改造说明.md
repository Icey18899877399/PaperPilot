## 一、US-02 生成论文智能导读

**改造前问题：** 导读只有概述+3要点+3思考题，缺少验收标准要求的九项结构；无页码引用；Prompt无"原文未说明"与防虚构约束；上下文只取论文开头8个切片（结论章节不在上下文内）；模型服务异常时接口直接500（提取式兜底被异常路径绕过）。

**修改文件与内容：**

| 文件 | 修改 |
|---|---|
| `backend/app/models/schemas.py` | 新增 `GuideCitation`、`GuideSection`；`GuideResponse` 扩展 one_liner、background、research_question、method、experiment、results、innovations、limitations、keywords、generator 字段，全部带默认值以兼容历史缓存 |
| `backend/app/agents/paper_agent.py` | ① 上下文改为全篇分节采样（每章节代表切片+首中尾+跳过参考文献），带`[章节·页码]`标注；② 重写系统Prompt：九项JSON输出、每项结论附章节页码引用、未说明项固定输出"原文未说明"、禁止虚构数值/数据集/对比结果；③ `complete_json` 用 try/except 包裹，`LLMServiceError` 时进入提取式降级（generator="fallback"）而非500；④ 越界页码引用视为幻觉丢弃；⑤ 日志补充生成模式与耗时 |
| `backend/app/api/routes/papers.py` | `/guide` 接口对残余 `LLMServiceError` 返回结构化502 |
| `frontend/src/types.ts`、`components/GuidePanel.tsx`、`App.tsx`、`styles.css` | 九项分区渲染+关键词标签+降级模式提示；引用渲染为"章节·第N页"可点击标签，点击经 `targetCitation` 跳转PDF对应页；旧版缓存导读仍按旧版布局渲染 |

**修改文件与内容：**

## 二点九、导读提示词三层重构与JSON稳健性（2026-07-15 第五轮，按用户反馈）

**问题：** 三个版本（通用学术/CS·AI/初学者）产出几乎无差别。根因是"每字段固定句数/句式"写死在共享契约里，刚性规则把版本笔法抹平了。

**提示词三层重构（`backend/app/prompts/guide.py` 整体重写）：**
- `GUIDE_SCHEMA_CONTRACT`（不可改）：只固定JSON结构、字段名、引用规则、防虚构红线——`paper_agent` 解析依赖它；
- `GUIDE_FIELD_STANDARDS`（共享）：只规定各字段"必须覆盖什么内容"，不规定篇幅笔法，给版本差异留空间；
- 每个版本新增 `directives`（专属笔法）：术语策略、深度侧重、篇幅、思考题风格、语气示例——这是拉开区分度的关键。
- `system_prompt` 拼接顺序：版本视角 → 版本笔法 → 共享字段标准 → 输出格式与红线（红线置末强调）。
- `GuidePromptVersion` 新增 `directives` 字段；对外符号（`get_guide_prompt`/`list_guide_prompts`/`GUIDE_PROMPT_REGISTRY`/`DEFAULT_GUIDE_PROMPT_KEY`/`system_prompt`）签名不变，路由与Agent无需改动。

**实测三版差异（同一篇论文）：** CS·AI版思考题为消融式"移除LRAE检索步骤评估会如何变化"、暴露 Llama3-8b-SFT 等工程细节；初学者版思考题为直觉式"为什么开放式比封闭式更难"、把数字翻译成含义；通用学术版问方法论严谨性、中性均衡。

**JSON稳健性加固（`backend/app/services/llm.py`）：** 发现即便开启 json_object 模式，DeepSeek 仍偶发返回带代码围栏或前后夹带文字的内容，导致整篇导读被打成降级（默认版本一次连挂两次）。新增 `_parse_json_object`：原样解析失败后依次剥离Markdown围栏、截取最外层 `{…}` 子串重试。加固后默认版本连续3次生成均为 `llm` 模式。新增 `tests/test_llm_json.py`（5例）。

## 二点十、导读提示词五点润色（2026-07-15 第六轮，按用户反馈）

仅改 `backend/app/prompts/guide.py`（不碰解析逻辑）：
1. **全局语言规则**（加入 GUIDE_SCHEMA_CONTRACT）：叙述一律简体中文，仅专有名词/模型名/指标名保留英文——修复 CS·AI 版曾把 one_liner 整句写成英文的漂移。
2. **引用接地收紧**：citations 只能引用片段标注中实际出现过的完整"[章节·第N页]"组合，section 与 page 必须同源，禁止拼配或杜撰页码。
3. **补回 few-shot 示例**：SCHEMA 末尾放一个全字段示例对象（content 留白 + limitations 的"原文未说明"空引用范式），固化格式、降低 JSON 抖动。
4. **叙事连贯约束**（加入 GUIDE_FIELD_STANDARDS）：results 应回应 research_question，七字段构成"背景→问题→方法→实验→结果→创新→局限"连贯叙事。
5. **CS·AI 版语气示例**改为中文主干、英文术语内嵌，与规则1一致，避免示范 code-switching。

实测（同一篇论文，三版重新生成）：均为 `llm` 模式（无降级）；CS·AI 版 one_liner 恢复中文主干；三版引用页码全部落在 1~22，被引 8 个章节名全部命中论文真实章节（共29个）。默认 prompt 长度 1808→3021 字符。

## 二点十一、局限性字段"原文未说明"修复（2026-07-15 第七轮，按用户反馈）

**问题：** limitations 字段常年输出"原文未说明"。根因不在提示词，在采样——`paper_agent` 有两处截断都偏向丢弃靠后章节：① `_sample_guide_chunks` 先按页码排序再 `[:24]`，② 上下文 `"...".join(...)[:15000]` 按字符串尾部截断。局限/讨论/结论/未来工作章节多在论文靠后，两处叠加导致其证据进不了模型上下文。

**修改（仅 `backend/app/agents/paper_agent.py`）：**
- 新增 `LIMITATION_SECTION_KEYWORDS` 与 `_is_limitation_section`：识别 limitation/discussion/conclusion/future work 及中文对应章节；
- 重写 `_sample_guide_chunks`：先单独挑出这些章节的切片（每章节≤2个）放候选队首受保护，再"先占位后排序"合并（采样上限 24→26），确保靠后章节不被名额挤掉；论文无此类章节时 priority 为空，局限性仍如实留空；
- 上下文字符预算 15000→26000，容纳全部靠后切片，避免字符串尾部截断二次丢弃；

**实测（OpenForecast，本含 Limitations + 7 Conclusion 两章节）：** 采样命中2个局限/结论切片；limitations 从"原文未说明"变为真实内容"数据集构建依赖LLM性能且需大量计算资源；LRAE……需进一步提升鲁棒性"，引用 `Limitations·第9页`。新增采样单测2例（含靠后局限章节必被采到、无局限章节不强造）。

## 二点十二、US-04 检索"稳妥包"优化（2026-07-15 第八轮，按用户反馈）

仅改 `backend/app/services/knowledge_base.py` + 新增 1 个脚本：

1. **RRF 倒数排名融合**替换加权和（`search_scored`）：原 `0.55×keyword_norm + 0.45×vector_cos` 把"按最大值归一的关键词分"与"原始余弦"两种不可比尺度直接相加，排序易失真。改为两路各自排名、按 `权重/(K+rank)`（K=60）融合，只看排名不看分值绝对大小；两路都未命中的切片自然得0分被排除（保持"零命中返回空"）。
2. **索引期预计算切片词项**（`index()` 建 `_chunk_terms` 缓存，`remove()` 同步清理）：消除 `search_scored` 里对每个切片、每次查询重复分词的开销；`search_scored` 改为读缓存（未命中才即时算）。
3. **新增检索评测脚本** `backend/scripts/eval_retrieval.py`：对运行中的后端跑一组"问题→期望页码"，输出命中率 Hit@1/@3/@5 与 MRR，直接对齐验收标准"检索测试展示召回及相关度"。内置参考评测集面向英文论文 OpenForecast。

**实测**：单测 31 passed（RRF 未破坏既有检索用例）；评测脚本对参考论文得 Hit@1 83.3%、Hit@3 100%、MRR 0.9167；英文查询"LRAE retrieval augmented evaluation"稳定召回第5页 5 LLM-based Retrieval-Augmented Evaluation 为第1名。

**评测暴露的已知局限（记录备查）**：当前为关键词+本地TF-IDF的**词面检索**，无法跨语言匹配——中文问题问英文论文会大面积零召回（如"这篇论文有哪些局限性"→0条）。这正是优化方案第二梯队④（接入 embedding 语义检索）要解决的问题；评测脚本的语言须与论文语言一致，docstring 已注明。

## 二点十三、US-04 语义检索与持久化复用（2026-07-15 第九轮，进阶包①②）

**目标：** 打通中英跨语言语义检索（词面 TF-IDF 无法做到），并避免每次重启重复调用 embedding 接口。仅改 `backend/app/services/vector_index.py` + 测试；`.env` 加 embedding 配置。

**进阶包②·持久化复用（`vector_index.py`）：**
- 按"内容签名"（切片ID+内容的 sha256）持久化**稠密 embedding 向量**到 `data/indexes/<id>.vectors.json`；
- `index()` 改为：签名+模型都匹配则**直接复用持久化向量、不再调用 embedding 接口**；内容变化（重解析）签名失配则重新嵌入；
- 去掉原先 `index()` 开头无条件 `remove()`（会误删待复用文件），改为替换式覆盖，重解析仍不产生重复索引；
- 只持久化昂贵的稠密向量，稀疏 TF-IDF 每次内存重建。

**进阶包①·embedding 接入（`vector_index.py`）：**
- `/embeddings` 请求**分批**（EMBEDDING_BATCH_SIZE=32），避免超服务端条数限制；
- `__init__` 加可注入 `transport`，便于用 MockTransport 离线测试；
- 失败仍自动回退本地 TF-IDF。

**配置：** `.env` 追加 `EMBEDDING_API_URL=https://api.siliconflow.cn/v1`、`EMBEDDING_MODEL=BAAI/bge-m3`（多语种，中英跨语言强）、`EMBEDDING_API_KEY=<用户自填>`。

**发现并修复的缺陷（clobber）：** embedding 接口失败回退本地时，`index()` 仍会 `_persist()`，用空稠密向量覆盖已持久化的有效 embedding——一次网络抖动就会摧毁缓存、逼迫全量重嵌。已加保护：`remote_enabled` 但本次未拿到稠密向量时跳过持久化，不覆盖已有缓存。

**测试：** 新增 3 例（持久化+重启复用不重调；内容变则重嵌；故障不破坏可复用缓存）。全量 **34 passed**；未配置 embedding 时本地模式零回归（评测仍 Hit@1 83.3%/Hit@3 100%/MRR 0.9167）。

**真实联网验证（bge-m3 已配置）：**
- 跨语言打通：此前中文问英文论文大面积零召回，现"这篇论文有哪些局限性"→第9页 Limitations（rank1）、"实验用了哪些基线模型和评价指标"→第6页 6.1 Experimental Setup（rank1）；中文评测集 **Hit@1 66.7%、Hit@3 83.3%、MRR 0.7222**（对比词面基线约0%）。
- 复用生效：重启后 `KnowledgeBase.index` 复用全部 108 条持久化 bge-m3 向量、**0 次重新嵌入**；故障 transport 下持久化文件保持完好（108 条未被覆盖）。
- 持久化产物：`data/indexes/<id>.vectors.json` 存 108×1024 维 bge-m3 向量 + 内容签名 + 模型名。

## 二点十四、US-04 结果去重/多样性 MMR（2026-07-15 第十轮，按用户反馈）

**问题：** 重叠切分与同章节多切片会让 top-k 被近重复片段霸屏（如"LRAE 怎么工作"→连续3条都来自 6.3 Experiments on LRAE），聊天证据覆盖面受损。

**改动（仅 `backend/app/services/knowledge_base.py`）：**
- `search_scored` 融合排序后新增 **MMR（最大边际相关）重排**：先取宽候选池（≥limit×4），再贪心选取，每步最大化 `λ·相关度 −(1−λ)·与已选最大相似度`（`MMR_LAMBDA=0.7`）；候选不多于 limit 时按相关度原样返回；
- 新增 `_mmr_select` 与 `_chunk_similarity`；相似度 = 正文词项 Jaccard（复用 `_chunk_terms` 缓存，无额外重算）+ 同末级章节信号（0.5 权重，主压同章节 near-dup）；
- 首位仍是最相关切片，MMR 不牺牲 top-1 相关度。

**测试：** 新增 2 例（MMR 让 top-2 覆盖不同章节且首位不变；候选不足 limit 时原样返回）。全量 **36 passed**。

**实测（bge-m3 语义检索下）：** "LRAE检索增强评估方法"从 p8/p8/p8（1章节）变为 p8/p6/p18/p2（4个不同章节，且方法定义节"5 LLM-based Retrieval-Augmented Evaluation"被提到 rank2）；三条先前同章节霸屏的查询 top-4 均为4个不同章节；中文跨语言评测无回归（Hit@1 66.7%/Hit@3 83.3%，MRR 0.72→0.75）。
