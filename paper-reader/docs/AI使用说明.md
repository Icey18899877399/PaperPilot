# PaperPilot AI 使用说明

## 1. 说明范围

本文区分两类 AI 使用：

- 开发辅助：在需求拆分、代码编写、测试、重构和文档整理过程中使用的编程助手。
- 运行时能力：项目启动后实际调用的语言模型、视觉模型和文档解析模型。

本次核对覆盖仓库中的 65 个 Python 文件、17 个 TSX 文件、8 个 TypeScript 文件、4 个样式文件、7 个 PowerShell 脚本，以及 Docker、Vite、Nginx、依赖清单、工作流和项目文档。依赖锁文件只用于确认第三方依赖版本，不作为开发者代码归属判断依据。

代码本身通常不包含“由某工具生成”的可靠标记。统一格式、完整类型注解、模板化注释或相似测试结构可以说明代码可能经过自动化辅助，但无法单凭这些特征准确区分 Claude Code、GitHub Copilot、Codex 或其他工具。因此本文不再给每个文件强行指定生成工具，也不提供无法复核的精确占比。

## 2. 可确认的开发辅助情况

仓库原有的 AI 使用记录明确提到以下工具：

| 工具 | 原记录中的用途 | 当前可确认程度 |
| --- | --- | --- |
| Claude Code | 功能开发、重构和文档编写 | 有历史文档记录；提交记录未附工具元数据，无法逐文件复核 |
| GitHub Copilot | 代码补全、基础代码、脚本和测试 | 有历史文档记录；无法从代码风格与 Codex 准确区分 |
| Codex | 代码补全、基础代码、脚本和测试 | 有历史文档记录；无法从代码风格与 Copilot 准确区分 |

Git 提交记录能够确认作者、时间和变更内容，但不能确认编辑器中具体启用了哪个助手。当前源码也没有 `Generated by`、Co-authored-by AI 或工具专属签名。因此，对具体工具的描述应理解为项目开发过程的披露，而不是基于源码取证得到的确定结论。

## 3. 源码特征与辅助范围判断

### 3.1 较强的 AI 辅助特征

以下特征在多个模块中重复出现，说明项目主体很可能经过较大范围的编程助手参与：

- 后端大量采用统一的 `from __future__ import annotations`、类型注解、服务边界和异常包装。
- 复杂逻辑附近存在面向验收条款的长注释，如 `US-02`、`US-04`、`US-05`、`US-07` 和“进阶包”等标记。
- LLM、MinerU、embedding 和视觉服务均采用“外部服务失败后回退”的相似设计模板。
- 测试普遍使用 `Fake`、`Stub`、`MockTransport` 和 `monkeypatch` 隔离外部服务，结构较统一。
- 前后端类型、API 路径、流式事件和错误提示形成较完整的成对实现，变更覆盖面明显高于简单代码补全。
- 多份阶段说明和验收报告与实现中的用户故事编号互相对应，体现了按需求批量生成或整理实现材料的工作方式。

这些特征可以支持“存在 AI 辅助”的判断，但不能支持“某一行或某一文件必然由某个指定工具生成”的判断。

### 3.2 判断口径

下文使用三个等级描述工具归属：

| 等级 | 含义 |
| --- | --- |
| 明确记录 | 仓库原有 AI 使用说明直接写明了工具和文件或目录 |
| 综合判断 | 原记录给出了目录范围，当前文件又具有同批实现的结构与注释特征，但没有逐文件生成记录 |
| 无法确认 | 只能看出可能使用过编程助手，不能判断具体是 Copilot、Codex、Claude Code，或是人工编写 |

“明确记录”也不表示文件的每一行都由该工具生成。项目经过多次提交、分支合并和人工调整，较大的业务文件通常是共同编辑结果。

### 3.4 工具与目录的汇总关系

| AI 工具 | 有记录的主要目录或文件 | 适合采用的表述 |
| --- | --- | --- |
| GitHub Copilot / Codex | `backend/app/agents/` 基础层、`backend/app/services/` 基础层、主要 API、部分模型、后端脚本和测试、前端业务组件、早期项目文档 | “项目基础骨架和多项业务功能使用 GitHub Copilot 或 Codex 辅助开发” |
| Claude Code | `chat_agent.py`、`routes/chat.py`、`ChatPanel.tsx` 等聊天链路的历史改造，以及旧版部署与 AI 文档 | “聊天功能和部分文档曾使用 Claude Code 进行较大修改” |
| Codex | 当前 `docs/部署文档.md` 与 `docs/AI使用说明.md` 的代码核对和重写 | “最新部署与 AI 使用说明由 Codex 辅助整理，并根据仓库源码及测试结果复核” |
| DeepSeek | 不用于仓库文件归属；在项目运行时负责导读、问答、翻译和思维导图等生成任务 | “DeepSeek 是运行时模型，不等同于开发工具” |
| MinerU | `parser.py` 调用的 PDF 结构化解析服务 | “MinerU 是运行时解析工具，不应统计为编程助手生成代码” |

如果必须给出目录级归类，可以将主要后端、前端和测试代码写为“Copilot/Codex 辅助为主，Claude Code 参与聊天链路改造，随后由多名开发者继续合并与修改”。现有证据不支持把 Copilot 和 Codex 再拆成准确的文件数量。

### 3.5 不应据此认定为 AI 生成的内容

以下内容缺少足够证据，或本身主要来自工具链：

- `frontend/package-lock.json` 是 npm 根据依赖解析结果生成的锁文件，不属于编程助手生成代码。
- `.gitkeep`、包初始化文件和简短配置文件信息量过少，无法判断创作方式。
- 默认视频目录中的 JSON 和 SVG 资源不能仅凭格式判断使用了何种 AI。
- GitHub Actions、Dockerfile 和 Compose 文件写法较标准，既可能人工编写，也可能由模板或助手生成。
- CSS 中重复的主题规则可能来自多轮人工调样，也可能经过助手扩写，单凭重复度无法归属。

## 4. 代码审查后的模块总结

### 4.1 后端

后端采用 FastAPI。`Runtime` 在进程启动时组装存储、解析、检索、模型、视频、学习资源和 Agent 服务，并从 `backend/data` 恢复已保存论文及索引。

论文上传接口校验扩展名、MIME、文件大小和 PDF 文件头，以 SHA-256 阻止重复上传。解析可以在后台执行：配置 MinerU 时读取其 ZIP 结果并归一化正文、图片、图表、表格、公式、代码和列表；未配置时回退到 PyPDF 文本提取。

检索层同时维护关键词结果和向量结果。默认向量实现是本地 TF-IDF；配置远程 embedding 后可持久化稠密向量，失败时回退本地检索。知识库还包含正文连续块合并、展示区域分类、RRF 融合和 MMR 去重。

Agent 层由协调 Agent 路由至论文理解、翻译和聊天 Agent。导读采用九项结构化输出并保存引用；问答支持 SSE 流式返回，检索不到相关内容时直接提示证据不足；图片解释在视觉模型不可用时使用图题和上下文；翻译、导读和思维导图均有无模型降级路径。

数据使用 JSON 和文件目录持久化。论文删除会同步清理上传文件、切片、向量、导读、思维导图、翻译和解释缓存。当前没有用户认证、数据库事务或多实例并发写入机制，定位仍是单机课程项目。

### 4.2 前端

前端使用 React、TypeScript、Vite、react-pdf 和 markmap。主界面覆盖论文库、上传、PDF 阅读、结构化内容、智能导读、双语对照、思维导图、聊天、Agent 日志、视频库和扩展学习。

开发环境通过 Vite 代理 `/api` 与 `/media`；容器环境由 Nginx 代理到后端。`VITE_DEMO_MODE=true` 时改用浏览器内置示例数据，供 GitHub Pages 展示，不会上传本地 PDF。聊天和导读使用自定义 SSE 客户端，切换论文或离开页面时取消未完成请求。聊天历史保存在浏览器 `localStorage`，服务端没有独立的会话表。

### 4.3 测试与运维

后端测试覆盖健康检查、上传解析、MinerU 适配、切片合并、检索、向量缓存、导读、双语页面、流式 LLM、聊天、视觉解释、学习资源和视频模块。脚本提供 DeepSeek 连接测试、MinerU 健康检查、参考论文烟雾测试、持久化与删除测试等辅助流程。

Docker Compose 只编排 PaperPilot 前后端；MinerU 位于仓库根目录的独立 Compose 项目。GitHub Pages 工作流仅构建演示模式前端，不部署 FastAPI 后端。

## 5. 运行时 AI 与模型服务

| 能力 | 实际实现 | 是否必需 | 未配置或失败时的行为 |
| --- | --- | --- | --- |
| 论文导读、问答、翻译、思维导图 | DeepSeek 或其他 OpenAI Chat Completions 兼容模型 | 否 | 使用提取式结果或待配置提示，部分生成能力受限 |
| 图片与图表解释 | OpenAI 兼容多模态模型 | 否 | 使用 MinerU 图题、表格文本和相邻正文解释 |
| 语义向量检索 | OpenAI Embeddings 兼容服务 | 否 | 回退本地 TF-IDF |
| PDF 结构化解析 | MinerU 3.4 pipeline | 否 | 回退 PyPDF，只保留按页纯文本 |
| 学术资源检索 | OpenAlex、Crossref 等公开接口 | 否 | 返回其他可用来源或本地结果 |

DeepSeek 在本项目中通过兼容接口调用，不包含训练或微调。API 密钥只从 `.env` 读取，模型状态接口不会返回密钥；模型错误信息会进行基础脱敏。

MinerU 属于文档解析模型和工具链，不是聊天模型。它负责把 PDF 转成结构块和资源文件，为后续检索与生成提供输入。

## 6. 人工核验与使用边界

开发辅助生成的代码在进入主分支前应至少完成以下核验：

1. 后端运行 `python -m pytest tests -v`，前端运行 `npm run build`。
2. 使用真实 PDF 检查页码、表格、公式、图片路径和引用跳转，不只依赖模拟数据。
3. 对导读、翻译和问答抽查原文依据，模型输出不能替代论文事实核验。
4. 外部接口调用应检查超时、额度、隐私政策和数据出境要求。
5. API 密钥不得提交到 Git、写入测试样例或出现在截图和日志中。
6. 升级模型名称、MinerU 或前端依赖后重新执行完整回归测试。

项目当前没有自动保存开发提示词、助手对话记录或逐行生成来源。若后续需要精确审计，应在提交说明中记录使用的工具、人工修改范围和验证结果，或保留团队认可的开发日志；不应事后仅靠代码风格反推具体工具。
